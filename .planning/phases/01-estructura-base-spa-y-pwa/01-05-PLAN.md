# Plan: 01-05 — CSS base (custom properties, reset, layout, nav)

**Phase:** 1 — Estructura base, SPA y PWA
**Plan:** 01-05
**Goal:** Reemplazar los estilos placeholder de `index.html` con el CSS base definitivo de AutoValeur MA: custom properties completas (paleta, espaciado, radios), reset minimalista, layout mobile-first centrado en 480px, header de app, barra de navegación inferior fija con safe-area-inset, transiciones suaves entre páginas, y corrección de scroll en Android.

## Context

Este es el último plan de la Fase 1. Reemplaza el bloque `<style>` placeholder que los planes anteriores dejaron en `index.html` con el CSS definitivo que usarán todas las fases siguientes.

El CSS establece el sistema de diseño completo: custom properties como única fuente de verdad para colores, espaciado y radios. Mobile-first con `max-width: 480px` centrado. Barra nav inferior con `env(safe-area-inset-bottom, 0px)` para iPhone. Transiciones de página con `fadeIn` para eliminar el flash al navegar. Corrección de `scroll-padding-bottom` para que los teclados virtuales de Android no tapen los inputs (relevante desde la Fase 4).

El CSS también incluye las propiedades `--border` y `--nav-height` que otros planes (01-04 ya las usa) necesitan. Es importante mantener un orden claro: variables → reset → layout → components.

**Requisitos implementados:** PWA-05 (responsive 360px+), NAV-02 (nav inferior fija y correcta).

## must_haves

- [ ] El bloque `<style>` en `index.html` usa las custom properties definidas en `:root` (sin valores hardcoded para colores ni espaciados)
- [ ] La app es correctamente usable en una pantalla de 360px de ancho (probado con DevTools → Device Mode → Moto G4 360px)
- [ ] La barra de navegación inferior es fija y visible al hacer scroll hacia abajo en una página larga
- [ ] El contenido de las páginas no queda oculto detrás de la nav inferior (padding-bottom correcto)
- [ ] La transición entre páginas muestra un fadeIn suave (sin flash blanco abrupto)
- [ ] No hay scroll horizontal en 360px (overflow-x: hidden en body)

## Notes

- **`dvh` vs `vh`**: `100dvh` es la altura dinámica del viewport que se ajusta cuando la barra del browser de Chrome Android aparece/desaparece. `100vh` en Chrome Android no se ajusta y puede causar que el contenido quede tapado. El fallback `min-height: 100vh` antes de `100dvh` es necesario para Chrome < 108 (aunque este segmento es pequeño en 2026).
- **`overscroll-behavior: none` en `html`**: Elimina el rebote elástico de scroll en iOS que puede ser confuso en una PWA. Si en fases futuras se necesita scroll elástico dentro de algún componente específico, se puede sobreescribir con `overscroll-behavior: auto` en ese elemento.
- **`-webkit-tap-highlight-color: transparent`**: Elimina el flash azul/gris que aparece en móvil al tocar enlaces y botones. Es un detalle de pulido que mejora significativamente la sensación de app nativa.
- **`.app-header` y sticky**: El `position: sticky` del header requiere que el elemento padre no tenga `overflow: hidden`. Si en fases futuras se añade `overflow: hidden` a `main` o `body`, el sticky del header dejará de funcionar.
- **CSS logical properties**: Este CSS base usa propiedades físicas (`left`, `right`, `bottom`) para el layout de la nav. Las fases 2+ (i18n/RTL) las convertirán a logical properties (`inset-inline-start`, etc.) según el plan 02-04.
- **`.card`, `.btn-primary`, `.btn-secondary`**: Definidas aquí como utilidades base para que estén disponibles desde la Fase 4 sin necesidad de modificar el bloque `<style>` principal.
